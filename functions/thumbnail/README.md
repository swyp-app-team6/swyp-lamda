## 문제 해결 전체 정리

### 1. 확장자 없는 파일 썸네일 생성 실패

**문제**: `original/68-64f42d94-...` 처럼 확장자 없는 키로 업로드된 파일이 썸네일 생성 없이 스킵됨

**원인**: Lambda가 파일명 확장자만으로 이미지 포맷을 사전 판단 → 확장자 없으면 무조건 스킵

**🔑 의사결정**: 확장자/Content-Type 기반 사전 필터링을 아예 제거하고, `sharp`가 실제 바이너리를 읽어 자동으로 포맷을 판별하게 함. 처리 실패 시에만 catch에서 "이미지가 아니었다"고 판단.
→ **근거**: 파일명이나 메타데이터는 신뢰할 수 없는 힌트일 뿐, 실제 이미지 여부의 유일한 진실은 바이트 내용 자체이기 때문

---

### 2. Content-Type 설정 책임 (프론트/백엔드)

**논의**: presigned URL 업로드 시 Content-Type을 누가 정확히 설정해야 하는가

**🔑 의사결정**: 백엔드에서 `image/jpeg` 하드코딩 금지, 프론트가 실제 파일의 MIME 타입을 감지해서 백엔드로 전달 → 백엔드는 받은 값을 그대로 서명에 사용
→ **근거**: 실제 업로드 파일이 JPEG가 아닐 수 있음(PNG, WebP 등). 다만 **이건 Lambda 버그의 직접 해결책이 아니라 UX/캐싱 목적의 별도 개선**으로 분리해서 판단함

---

### 3. 재인코딩 후 용량이 오히려 커짐

**문제**: 클라이언트가 이미 압축해서 올린 원본을 Lambda가 처리한 후 용량이 더 커짐

**원인**: `fit: "cover"` 리사이즈에 `withoutEnlargement` 옵션이 빠져서, 원본이 목표 크기보다 작을 때 **강제로 확대(upscale)**되며 압축 효율이 떨어짐

**🔑 의사결정**: `withoutEnlargement: true` 적용으로 확대 자체를 차단
→ 처음엔 metadata로 `needsUpscale`을 수동 체크하는 방식으로 구현했으나, 이후 **sharp 옵션 하나로 대체 가능**하다는 게 드러나 로직 단순화 (5번 항목과 연결)

---

### 4. 보관함/메인 뷰 — 2개 사이즈 요구사항 반영

**요구사항**: 보관함 208×220, 메인 뷰 284×392, 업로드 이미지 타입은 사전에 알 수 없음

**🔑 의사결정 A — 한 원본에서 두 variant를 각각 독립적으로 생성**
- 대안이었던 "메인을 먼저 만들고 거기서 보관함을 파생"은 **실측으로 기각**함
- 근거: 두 비율 차이가 약 23%(0.945 vs 0.724)라 2차 크롭 시 얼굴이 추가로 잘리는 구도 손실이 실제 샘플에서 확인됨 → 계산 비용은 늘지만 품질 우선으로 원본에서 각각 크롭하는 쪽 채택

**🔑 의사결정 B — 크롭 포지션: `attention`(자동 saliency) → `north`(상단 기준)으로 변경**
- 근거: 실제 프로필 사진 3장 테스트에서 `attention`이 얼굴을 크롭 범위 밖으로 미는 경우 발견, `north`가 가장 안정적
- 단, **완전한 해결책은 아님**을 인지 — 셀카 각도/반신샷 등 예외 케이스는 여전히 남아있어, 배포 후 신고가 들어오면 얼굴 인식 기반(AWS Rekognition) 크롭으로 전환 검토하기로 함

**🔑 의사결정 C — 출력 포맷: WebP, quality 90**
- 실측: 비슷한 화질 구간에서 WebP가 JPEG(mozjpeg) 대비 30%+ 작음 확인
- quality 85 vs 90 체감 차이는 크지 않았으나, 용량 차이도 크지 않아 **화질 우선으로 90 선택**

---

### 5. 구현 방식: 버퍼 → 스트림(tee)으로 전환

**초기 구현**: 원본을 버퍼로 전부 받아 `sharp().clone()`으로 두 파이프라인 분기
- 당시 이유: metadata 사전 조회(`needsUpscale`, `hasAlpha` 판단), 재인코딩 결과와 원본 크기 비교 안전장치가 버퍼 없이는 구현이 번거로웠음

**재검토 계기**: "안전장치가 실측상 의미 없다"는 지적 → 필요성 재검토

**🔑 최종 의사결정 — 스트림 tee(PassThrough) 방식으로 전환**
- `needsUpscale` 수동 체크 → `withoutEnlargement: true`로 대체되어 metadata 사전조회 불필요
- `hasAlpha` 조건부 flatten → 알파 없는 이미지엔 무해한 no-op이므로 무조건 호출로 단순화
- "원본보다 커지면 원본 사용" 안전장치 → 실측상 항상 원본 대비 1% 미만이라 제거
- 결과적으로 metadata를 미리 알아야 할 이유가 모두 사라져, 버퍼 없이 청크 단위로 원본을 두 갈래로 흘려보내는 tee 구조로 재구현
- **인지하고 있는 잔여 리스크**: 두 분기 간 처리 속도 차이 시 backpressure 미처리, 부분 실패(하나만 실패) 시 세밀한 제어 없음 — 현재 규모(작은 이미지, variant 2개)에선 문제 안 될 것으로 판단, 배포 후 CloudWatch 지표로 확인 예정
